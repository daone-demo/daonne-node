# Daone 管理后台后端技术方案与表结构设计

> 页面来源：`https://admin.daoneai.com/`，已于 2026-06-23 重新读取。  
> 目标：在当前前台后端项目中补齐管理后台能力，方案尽量简单，但覆盖当前后台页面的完整功能。

## 1. 后台页面重新读取结论

当前后台实际页面如下：

| 页面 | 前端路由 | 页面功能 | 列表字段 | 页面标注接口 |
|---|---|---|---|---|
| 首页 | `#/welcome` | 运营概览、趋势、待办、快捷入口 | 累计用户、今日订单、今日流水、AI 调用量、业务趋势、待办事项 | 未标注接口，需要新增聚合接口 |
| 工作流列表 | `#/workflows/list` | 管理可配置工作流 | 工作流、场景、节点数、创建人、更新时间、状态 | `GET /v1/workflows（暂无 /admin 接口）`，待建设 |
| 用户列表 | `#/users/list` | 用户、会员、积分、项目使用情况 | 用户昵称、手机号、邮箱、角色、注册时间、状态 | `GET /admin/v1/users` |
| 开票列表 | `#/invoices/list` | 审核开票申请，维护开票状态 | 申请编号、发票抬头、关联订单、类型、金额、申请时间、状态 | 暂未提供 `/admin` 开票接口，待建设 |
| 模型列表 | `#/models/list` | AI 能力、任务类型、积分成本、调用状态 | 模型名称、模型编码、任务类型、基础积分、今日调用、更新时间、状态 | `GET /admin/v1/model-configs` |
| 套餐列表 | `#/plans/list` | 会员套餐、计费周期、价格、权益 | 套餐、套餐编码、价格配置、套餐权益、状态 | `GET/POST /admin/v1/plans` |
| 灵感发现 | `#/content/inspirations` | 前台灵感瀑布流内容运营 | 灵感内容、分类编码、封面 URL、创作提示词、更新时间、状态 | `GET/POST /admin/v1/inspirations` |
| 分类管理 | `#/content/categories` | 灵感和模板内容分类 | 分类名称、分类编码、使用范围、内容数、排序、状态 | 暂未提供 `/admin` 分类接口，待建设 |
| 提示词模板 | `#/content/prompts` | AI 场景提示词模板 | 模板名称、模板编码、使用场景、模板内容、更新时间、状态 | `GET/POST /admin/v1/prompt-templates` |
| 订单 | `#/finance/orders` | 套餐订单、支付渠道、金额、交易状态 | 订单号、用户 ID、商品、金额、支付方式、下单时间、状态 | `GET /admin/v1/orders` |

结论：后台页面本身是一个运营管理壳，当前只需要补齐工作流、开票、分类、首页聚合，并完善已有用户、模型、套餐、灵感、提示词、订单接口即可。

## 2. 设计原则

1. 继续使用当前 Node/Vercel 后端，不新增独立后台服务。
2. 后台 API 统一前缀：`/api/admin/v1`。
3. 鉴权继续复用当前 `Authorization: Bearer <token>`，并要求当前用户 `role=ADMIN`。
4. 首页指标优先实时聚合，不单独建统计快照表，保持简单。
5. 后台写操作统一记录操作日志。
6. 优先复用前台已有表，只有当前表结构没有自然落点时才新增表。
7. 新增表统一包含基础字段：`id`、`gmt_create`、`gmt_modified`、`deleted`、`attributes`。

## 3. 统一表字段约定

新增表统一包含以下字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `BIGINT` | 自增主键 |
| `gmt_create` | `TIMESTAMP` | 记录创建时间 |
| `gmt_modified` | `TIMESTAMP` | 记录修改时间 |
| `deleted` | `BOOLEAN` | 逻辑删除标记 |
| `attributes` | `JSON` | 扩展字段，放非核心、低频变化字段 |

Postgres 可使用 `BIGSERIAL` 或 `GENERATED BY DEFAULT AS IDENTITY`；MySQL 可使用 `BIGINT AUTO_INCREMENT`。下面 SQL 用通用写法表达，落库时按目标数据库微调。

已有前台业务表为了兼容现有接口，不建议立即重命名字段；后台接口层统一做字段映射：

| 后台统一字段 | 新增表 | 旧业务表兼容方式 |
|---|---|---|
| `id` | 自增主键 | 复用原 `id` |
| `gmt_create` | 直接落库 | 映射原 `created_at` |
| `gmt_modified` | 直接落库 | 映射原 `updated_at` |
| `deleted` | 直接落库 | 优先映射原 `deleted`，没有则由 `status` 推导 |
| `attributes` | 直接落库 | 缺失的旧表补 JSON 字段 |
 

## 4. 前台表复用判断

| 后台页面 | 复用表 | 复用方式 | 是否新增表 |
|---|---|---|---|
| 首页 | `user_account`、`payment_order`、`generation_task`、`invoice_application`、`workflow` | 实时聚合统计 | 否 |
| 工作流列表 | `workflow` | 复用前台工作流表，`attributes.adminTemplate=true` 表示后台模板，`attributes.categoryCode/nodeCount/ownerName/status/workflowCode` 存后台字段 | 否 |
| 用户列表 | `user_account`、`point_account`、`user_subscription`、`project` | 列表和详情聚合 | 否 |
| 开票列表 | 无现成表 | 订单表只保存支付订单，不适合承载发票抬头、税号、开票状态 | 是 |
| 模型列表 | `model_config`、`generation_task` | 模型配置复用 `model_config`，今日调用从 `generation_task` 聚合 | 否 |
| 套餐列表 | `subscription_plan`、`subscription_plan_price` | 套餐和价格配置直接复用 | 否 |
| 灵感发现 | `inspiration` | 复用灵感表，补 `prompt/deleted/attributes` | 否 |
| 分类管理 | 无现成表 | 当前只有内容上的 `category_code` 字段，没有分类名称、范围、排序、状态 | 是 |
| 提示词模板 | `prompt_template` | 直接复用，补 `deleted/attributes` | 否 |
| 订单 | `payment_order`、`payment_transaction` | 订单和支付交易直接复用 | 否 |

最终建议：只新增 `invoice_application`、`content_category`、`admin_operation_log` 三张表；工作流模板不再单独建 `admin_workflow_template`。

## 5. 最小新增表结构

### 5.1 开票申请表

用于后台“开票列表”。

```sql
CREATE TABLE invoice_application (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    invoice_no VARCHAR(40) NOT NULL,
    user_id BIGINT NOT NULL,
    order_no VARCHAR(40) NOT NULL,
    invoice_title VARCHAR(128) NOT NULL,
    tax_no VARCHAR(64) NOT NULL,
    invoice_type VARCHAR(32) NOT NULL,
    amount_fen BIGINT NOT NULL,
    status VARCHAR(16) NOT NULL DEFAULT 'PENDING',
    email VARCHAR(128),
    receiver_name VARCHAR(64),
    receiver_phone VARCHAR(20),
    receiver_address VARCHAR(500),
    reject_reason VARCHAR(500),
    issued_at TIMESTAMP,
    invoice_file_asset_id BIGINT,
    express_company VARCHAR(64),
    express_no VARCHAR(64),
    gmt_create TIMESTAMP NOT NULL,
    gmt_modified TIMESTAMP NOT NULL,
    deleted BOOLEAN NOT NULL DEFAULT FALSE,
    attributes JSON,
    CONSTRAINT uk_invoice_no UNIQUE (invoice_no)
);

CREATE INDEX idx_invoice_order ON invoice_application(order_no);
CREATE INDEX idx_invoice_user_create ON invoice_application(user_id, gmt_create);
CREATE INDEX idx_invoice_status_create ON invoice_application(status, gmt_create);
```

发票类型：`VAT_NORMAL`、`VAT_SPECIAL`。  
状态：`PENDING`、`PROCESSING`、`ISSUED`、`REJECTED`、`CANCELLED`。

### 5.2 内容分类表

用于后台“分类管理”，同时服务灵感发现和模板/工作流分类。

```sql
CREATE TABLE content_category (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    category_code VARCHAR(32) NOT NULL,
    category_name VARCHAR(64) NOT NULL,
    scope VARCHAR(32) NOT NULL,
    parent_code VARCHAR(32),
    sort_no INT NOT NULL DEFAULT 0,
    status VARCHAR(16) NOT NULL DEFAULT 'ENABLED',
    gmt_create TIMESTAMP NOT NULL,
    gmt_modified TIMESTAMP NOT NULL,
    deleted BOOLEAN NOT NULL DEFAULT FALSE,
    attributes JSON,
    CONSTRAINT uk_content_category_code UNIQUE (category_code)
);

CREATE INDEX idx_content_category_scope_status
    ON content_category(scope, status, sort_no);
```

使用范围：`ALL`、`INSPIRATION`、`TEMPLATE`。  
状态：`ENABLED`、`DISABLED`。

### 5.3 后台操作日志表

用于记录后台所有写操作，支撑追责和问题排查。

```sql
CREATE TABLE admin_operation_log (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    admin_user_id BIGINT NOT NULL,
    action VARCHAR(64) NOT NULL,
    target_type VARCHAR(64) NOT NULL,
    target_id VARCHAR(64),
    request_method VARCHAR(10),
    request_path VARCHAR(255),
    before_json JSON,
    after_json JSON,
    remark VARCHAR(500),
    ip VARCHAR(64),
    user_agent VARCHAR(500),
    gmt_create TIMESTAMP NOT NULL,
    gmt_modified TIMESTAMP NOT NULL,
    deleted BOOLEAN NOT NULL DEFAULT FALSE,
    attributes JSON
);

CREATE INDEX idx_admin_operation_admin_create
    ON admin_operation_log(admin_user_id, gmt_create);
CREATE INDEX idx_admin_operation_target
    ON admin_operation_log(target_type, target_id);
```

## 6. 复用表的最小改造

为了让后台页面完整显示，现有表只做必要扩展，不重复建表。旧表已有 `id/created_at/updated_at/deleted` 的，后台接口层映射为 `id/gmt_create/gmt_modified/deleted`。

### 6.1 `workflow`

复用为后台工作流模板表，新增 `attributes` 后用 JSON 存放后台专属字段：

```sql
ALTER TABLE workflow ADD attributes JSON;
```

后台模板筛选规则：

```text
deleted = false
AND attributes.adminTemplate = true
```

`attributes` 建议结构：

```json
{
  "adminTemplate": true,
  "workflowCode": "WF-1024",
  "categoryCode": "ECOMMERCE",
  "categoryName": "电商营销",
  "nodeCount": 12,
  "ownerName": "运营管理员",
  "status": "ENABLED"
}
```

### 6.2 `inspiration`

后台灵感页面有“创作提示词”，当前表需要补字段：

```sql
ALTER TABLE inspiration ADD prompt JSON;
ALTER TABLE inspiration ADD attributes JSON;
ALTER TABLE inspiration ADD deleted BOOLEAN NOT NULL DEFAULT FALSE;
```

如果后续做新版本表结构，建议把 `created_at/updated_at` 统一调整或映射为 `gmt_create/gmt_modified`。当前为了降低改造成本，可在 API 层把已有 `created_at`、`updated_at` 映射为前端需要的时间字段。

### 6.3 `prompt_template`、`model_config`、`subscription_plan`

这些表当前已经能支撑页面主体功能。为满足统一扩展能力，建议补：

```sql
ALTER TABLE prompt_template ADD attributes JSON;
ALTER TABLE prompt_template ADD deleted BOOLEAN NOT NULL DEFAULT FALSE;

ALTER TABLE model_config ADD attributes JSON;
ALTER TABLE model_config ADD deleted BOOLEAN NOT NULL DEFAULT FALSE;

ALTER TABLE subscription_plan ADD attributes JSON;
ALTER TABLE subscription_plan ADD deleted BOOLEAN NOT NULL DEFAULT FALSE;
```

若做一次完整 V2 迁移，再统一补 `gmt_create/gmt_modified` 或把旧字段重命名。短期不强制重命名，避免影响前台接口。

### 6.4 其他直接复用表

| 表 | 后台用途 | 是否改表 |
|---|---|---|
| `user_account` | 用户列表、管理员账号、角色状态 | 否 |
| `point_account` | 用户积分余额 | 否 |
| `point_ledger` | 用户积分流水、人工调账记录 | 否 |
| `project` | 用户详情中的项目数量和最近项目 | 否 |
| `payment_order` | 订单列表、首页今日订单和流水 | 否 |
| `payment_transaction` | 订单详情里的支付方式和交易信息 | 否 |
| `subscription_plan_price` | 套餐价格配置 | 否 |
| `generation_task` | 首页 AI 调用量、模型今日调用 | 否 |

## 8. 状态流转

### 8.1 开票

```text
PENDING -> PROCESSING -> ISSUED
PENDING -> REJECTED
PROCESSING -> REJECTED
```

约束：

1. 只有已支付订单允许开票。
2. 开票金额不得超过订单实付金额。
3. 一期建议一单一票，避免拆票复杂度。

### 8.2 工作流

```text
ENABLED <-> DISABLED
DELETE => deleted = true
```

约束：

1. `nodeCount` 由后端根据 `workflow_json` 计算后写入 `attributes.nodeCount`。
2. 删除使用逻辑删除。

## 9. 实施顺序

1. 新增表：`invoice_application`、`content_category`、`admin_operation_log`。
2. 补已有表字段：`workflow.attributes`、`inspiration.prompt`、`attributes`、`deleted`。
3. 增加 `/api/admin/v1/dashboard` 聚合接口。
4. 增加工作流、开票、分类接口。
5. 完善已有用户、模型、套餐、灵感、提示词、订单接口的详情、筛选、状态开关。
6. 所有后台写操作写入 `admin_operation_log`。

## 10. 对当前项目的改造点

| 文件 | 改造内容 |
|---|---|
| `src/starter/app.js` | 注册新增 `/api/admin/v1` 路由 |
| `src/service/admin/adminService.js` | 独立后台管理服务包，和前台业务服务区分 |
| `src/infrastructure/db/memoryStore.js` | 增加开票、分类、操作日志的内存集合和 seed；工作流继续用 `workflows` |
| `docs/sql/*.sql` | 同步新增表和必要 ALTER |
| `src/starter/openapi.js` | 补充后台接口 OpenAPI |
| `tests/api.test.mjs` | 增加后台接口基础测试 |

这样设计后，新增表从 4 张减少为 3 张，工作流、用户、订单、套餐、模型、提示词、灵感、AI 调用统计都复用前台已有表，接口覆盖当前后台页面，同时避免提前引入复杂的权限模型、统计快照或后台独立服务。
